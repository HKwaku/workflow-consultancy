Vesno (Workflow Consultancy) - Architecture

AI-native operating-model + M&A diligence platform · Living-workspace model · Last updated 2026-05-13

Living-workspace contract

  • The process IS the deliverable. There are no separate "report", "redesign", "analysis", or "export" artefacts. The canvas + chat is what ships to the user.
  • Cost, savings, and automation metrics derive on read from processes.flow_data.rawProcesses[].steps[] via lib/processMetrics.js. No cached snapshot columns.
  • AI improvement suggestions land as inline rows in the changes table on the live process. Users apply them in place; there is no separate "save the redesign" step.
  • Deal findings live directly on deal_findings (deal_id, finding_key). No per-analysis-run snapshot scope.
  • Every server mutation routes through the changes relational audit log.

Client journey

End-to-end path a customer takes through the product, and the surfaces each stage touches. Each step maps onto the layers in the diagram below.

  1. 1 · Discover & sign in
    Marketing site (app/page.jsx → MarketingClient, statically renderable for SEO) with a CTA into /workspace/map. Auth is Supabase (/signin, components/auth/SignInForm.jsx). A returning signed-in user with an operating model is redirected straight past marketing into their active model (cheap /api/me/operating-model probe → resolveDefaultModelForUser: per-member preferred → org default; ?stay=1 opts out).
  2. 2 · Map the first process (onboarding)
    The chat-driven canvas (/process-audit / /workspace/map, DiagnosticWorkspace.jsx). Reina (process agent, ALL_CHAT_TOOLS) walks the user through capturing real steps, owners, systems, volumes, and costs. Each turn upserts processes.flow_data via PUT /api/update-diagnostic; cost / savings / automation derive on read. No "generate report" step - the canvas IS the deliverable.
  3. 3 · Build out the operating model
    Processes are filed under functions; roles and systems form the inventory. The model agent (MODEL_AGENT_TOOLS) answers model-level questions and proposes function / role / system and process-lifecycle changes as Confirm cards (nothing writes until the user clicks). Analytics + Insights tabs roll the model up. An org can hold multiple operating models; the Standard-scope WorkspaceModelsTab creates / switches them and persists the member's active model (organization_members.preferred_operating_model_id, migration 41) so Home / New chat follow it.
  4. 4 · Deal & data room (M&A / PE)
    Create a deal, invite participants, upload documents (Inngest processDealDocument: parse → chunk → embed). The deal agent (DEAL_AGENT_TOOLS) searches the room with citations; findings hang on deal_findings, with Q&A and threaded comments. Document visibility is role-scoped per participant.
  5. 5 · Produce keepable outputs
    The user asks Reina for a concrete deliverable; emit_artefact briefs the artefact sub-agent. It lands in workspace_artefacts, appears in the rail Artefacts slider (one unified list), and renders on the render-only Outputs canvas. Office files (.pptx/.docx/.xlsx) build asynchronously — they show as "Building…" and become downloadable on their own, so the user keeps working.
  6. 6 · Live, ongoing
    The workspace stays current. User and agent edits land as rows in the changes audit log and apply in place; metrics re-derive on every read, so the model never goes stale. GDPR data export and account erasure are available from rail settings (/api/me/* + the expunge cron).

Key data flows

  1. User edits a step on the canvas
    DiagnosticWorkspace → PUT /api/update-diagnostic → processes.flow_data upsert → diffStepsForChangelog() emits modified rows into the changes table. The metrics (total_annual_cost, etc.) are recomputed on the next read via deriveProcessMetrics.
  2. User chats with Reina
    Browser opens an SSE stream to POST /api/diagnostic-chat. pickAgent (router.js) selects one of three toolsets - process (ALL_CHAT_TOOLS, 70), model (MODEL_AGENT_TOOLS, 26), or deal (DEAL_AGENT_TOOLS, 16) - and the graph in lib/agents/chat/graph.js drives an Anthropic tool-use loop. Open-process canvas edits apply inline. Everything outside the open canvas - workspace adds, process lifecycle (create/duplicate/file/delete_process, Tier 1) and operating-model edit/delete (propose_update/move/delete_function|role|system, Tier 2) - stages a proposal and emits a deal_proposal/workspace_proposal SSE event; the client renders a Confirm card that, on click, fires the matching POST/PATCH/DELETE on /api/operating-models/[id]/…. Nothing is written server-side until that confirm.
  3. Reina produces a keepable deliverable
    The agent calls the emit_artefact tool with a skill + brief → the executor delegates to the artefact sub-agent (lib/agents/artefacts/generate.js, per-skill model tier, prompt-cached skill prompt, structured outputs for schema-bearing skills) → text/structured: the validated body is written to workspace_artefacts synchronously; office (.pptx/.docx/.xlsx): a "building" placeholder row is written and the buildOfficeArtefact Inngest worker fills it in async → an artefact SSE event fires; the mounted Outputs panel refreshes live (and polls while anything is "building") and renders by type (interactive Gantt, diagram, table, doc, downloadable office file).
  4. A deal document is uploaded
    POST /api/deals/[id]/documents → Supabase Storage + deal_documents insert (status=pending) → Inngest event deal-document.uploaded → processDealDocument worker: extract text → chunk → embed via Voyage → flip to ready. OCR fallback via Mistral when text extraction yields nothing for image-based PDFs.
  5. A user asks Reina to search the data room
    Agent calls search_deal_documents tool → lib/deal-analysis/chunkSearch.js embeds the query and runs the search_deal_chunks Postgres RPC (hybrid vector + keyword + ACL). Results render in chat with citations.
  6. An auditor reviews the deal page
    DealWorkspaceModal fetches /api/deals/[id], /documents, /qa, /checklist in parallel. Findings hang directly on the deal (deal_findings (deal_id, finding_key)) - no per-analysis-run snapshots.
  7. Auth on every request
    requireAuth peeks the JWT, validates locally on cache hit, and only round-trips to Supabase Auth on cache miss. Coalesces concurrent same-token validations into a single network call.

Outputs & artefact generation

Generated content that has no place in the canonical schema - tables, docs, code, datasets, diagrams, project plans - is persistent and model-scoped. There is one list and one render surface. The chat rail's Artefacts slider is the single list: persistent outputs (latest of each meta.supersedes lineage) sit alongside session snapshots/reports, one unified list however produced. Selecting one fires vesno:open-workspace (scope:'outputs') + vesno:open-output-artefact and the Outputs canvas (/workspace?view=outputs, WorkspaceOutputsTab.jsx, mounted when effectiveCanvasScope==='outputs') renders just that artefact. The Outputs canvas is render-only - its old left-hand list/sidebar was removed (the rail slider owns the list) and it is a single flex column with one scroll region. A pending-selection ref covers the mount race when the tab opens from the same click. Emitting an artefact never mutates the model.

The chat emit_artefact tool does not write the body itself: it picks a skill and briefs a focused sub-agent (lib/agents/artefacts/generate.js) that runs one-shot on a per-skill model tier (most text/table/diagram/code → Haiku 4.5 for speed; heavy-synthesis skills → Opus 4.7; office → Sonnet 4.6), with the static per-skill system prompt prompt-cached and structured outputs (output_config.format) for skills that ship a jsonSchema; the rest use a validate→repair-once loop. Text/structured artefacts generate synchronously in the chat turn. Office skills (.pptx/.docx/.xlsx) build asynchronously: emit_artefact creates a placeholder row (meta.build.status='building'), returns to the chat immediately, and queues the buildOfficeArtefact Inngest worker, which builds the file in the code-execution sandbox (python-pptx / python-docx / openpyxl, base64-over-stdout) and stores the bytes in a private Supabase Storage bucket (migration #40) fronted by a member-gated /file route; the Outputs panel polls and the finished file appears on its own. A failed build flips to build.status='failed' and the UI offers regenerate — never a JSON body saved as .pptx. Without Inngest, emit_artefact falls back to an inline synchronous build. The skill registry is lib/agents/artefacts/skills.js (47 skills + the special raw = 48 enum values); text artefacts are stored in workspace_artefacts (migration #39, member RLS, version lineage via meta.supersedes, pointer to any binary in meta.file); helpers in lib/operatingModel/artefacts.js; CRUD at /api/operating-models/[id]/artefacts. Adding a skill is one registry entry + a validator - it auto-registers across every agent that has the tool.

Production checklist

  1. All required env vars set in Vercel; NEXT_PUBLIC_APP_URL set for CORS / Origin checks.
  2. Migrations applied via DATABASE_URL=… npm run migrate (idempotent; records to schema_migrations). This covers the three living-workspace files (migration-living-workspace-1-schema.sql, -2-rls.sql, -3-drop-compat.sql) and the artefact store. Legacy scripts/ files already applied on the original install should be inserted into schema_migrations manually so the runner skips them - see supabase/MIGRATIONS.md.
  3. Inngest project synced (/api/inngest reachable, INNGEST_EVENT_KEY set).
  4. RLS policies pass smoke tests (a non-member cannot read a deal or process they don't own).
  5. Rate limit defaults reviewed (lib/rate-limit.js); upgrade to Redis if you need shared limits across instances.
  6. .env.local never committed; secrets only in Vercel.
  7. Mobile: app/layout.jsx exports viewport = { width:'device-width', initialScale:1, viewportFit:'cover' } — without it the responsive layer (~30 @media blocks, primary max-width:768px, in public/styles/diagnostic.css) is inert on phones. viewport-fit=cover activates the existing env(safe-area-inset-*) notch handling.
